# 環境依存フレーキーの落とし穴集

「ローカルは緑・CIは赤」「再実行すると結果が揺れる」テストは、コードのバグではなく**実行環境の差**が
原因のことが多い。ローカル(例: macOS)と CI(例: Linux + コンテナ化された DB) では OS・DB・
タイムゾーン・実行順・数値精度が違い、テストがその差に暗黙に依存していると環境ごとに結果が変わる。

各パターンを「兆候 → なぜ起きる → 対処」で並べる。まず自分の症状に近いものを探す。
記述は言語・フレームワーク非依存の概念で書いてある。自分の言語・テストランナー・ORM の
対応する機能（時刻固定・許容誤差比較・順序非依存の比較など）に読み替えて適用する。

## 1. 時刻の精度差（ナノ秒 vs マイクロ秒）

**兆候**: 更新時刻等のタイムスタンプを厳密等値（`==` 相当）で比較するテストが、ローカルでは
通るのに CI で**下位桁だけ違って**（例: `…906223609` vs `…906223000`）落ちる。再実行すると
通ったり落ちたりする。

**なぜ起きる**: 言語ランタイムがメモリ上で持つ時刻型は高精度（ナノ秒等）だが、RDB の timestamp
列は精度が低いことが多い（マイクロ秒精度で下位桁が切り捨てられる、等）。そのため

- 生成・構築直後の**メモリ上オブジェクト**のタイムスタンプ（高精度）と、
- 再取得した**永続化層（DB）由来**のタイムスタンプ（低精度）

を厳密等値で比べると、丸めた下位桁が一致せず落ちる。ローカルでたまたま高精度側の下位桁が 0
だと通り、CI で非0だと落ちるため、環境依存フレーキーに見える。

**対処**:
- 比較の**両辺を同じ由来に揃える**。基準値も比較値も両方 DB から再取得した値にする。
- あるいは許容誤差付きの比較（例: 1秒以内の差を許容）にする、秒単位に丸めて比較する、
  DB から再取得する経路を通して統一的に値を取る。
- 注意: 時刻固定機能（現在時刻をテスト内で固定する仕組み）は「時刻が進まない」ことは保証するが、
  **メモリ値とDB値の精度差そのものは解決しない**。時刻を固定しても両辺の由来が違えば落ちる。

## 2. 実行順序への依存

**兆候**: 実行順をランダム化するテストランナーで、順序を決める乱数シードが変わると落ちる。
単体では通るのに全体実行だと落ちる（逆も）。

**なぜ起きる**: あるテストが残したグローバル状態・共有変数・DB レコード・キャッシュに、
別のテストが暗黙に依存している。実行順が変わると前提が崩れる。

**対処**: 失敗時のシードを控えて同じシードを指定して再現する（多くのランナーは失敗時に再現用の
シード指定コマンドを出力する）。状態リークを断つ（各テストで必要なデータを自前で用意し、
共有データを書き換えない。グローバル状態はセットアップ/ティアダウンで確実にリセット）。
「先に走る別テストが作ったデータ」に依存していないか疑う。

## 3. タイムゾーン / ロケール差

**兆候**: 日付境界（月末・日付の変わり目）や曜日・数値/通貨フォーマットのテストが、CI でだけ
1日ずれる・表記が違う。

**なぜ起きる**: ローカルと CI で OS の TZ・locale 設定が違う。システム時刻をタイムゾーン指定なしで
素で使うと、実行環境のシステム TZ に引きずられる。

**対処**: アプリ側で明示的にタイムゾーンを指定した時刻を使い、テストは時刻固定機能で境界を避けた
具体時刻に固定する。フォーマット検証は locale を固定する。「今まさに月末」などの相対時刻に
依存したテストを書かない。

## 4. テスト並列分割の副作用

**兆候**: テストを複数ジョブ/プロセスに分割実行する構成（全体を N 分割し、そのうち i 番目を
各ジョブが担当する、といった分割）で、特定の分割でだけ落ちる。分割の切り方が変わると症状が動く。

**なぜ起きる**: 分割によって実行される組み合わせ・順序が変わり、順序依存（項目2）や共有リソース
（同じ外部ポート・固定ファイル名・DBのシーケンス値）の衝突が表面化する。

**対処**: 根っこは順序依存と共有リソース。各テストを独立させ、プロセス間で衝突する固定リソース
（ポート・一時ファイル名・外部サービス）を使わない。まず単一プロセスで再現するか切り分ける。

## 5. 浮動小数・集合の順序

**兆候**: 数値計算の厳密等値比較が末尾桁で落ちる。配列/コレクションの比較が「同じ要素なのに
順序違い」で落ちる。

**なぜ起きる**: 浮動小数の桁落ちは環境の演算順序で微妙に変わる。DB から取る集合の順序は
明示的な整列指定（`ORDER BY` 相当）がない限り保証されない。

**対処**: 数値は許容誤差付きで比較するか、固定小数（十進）型や明示的な丸めを使う。順序を問わない
比較は「順序を無視して要素集合として一致するか」を見る（順序非依存の比較・集合化して比べる）。
順序が意味を持つなら、明示的な整列指定をしてから検証する。
