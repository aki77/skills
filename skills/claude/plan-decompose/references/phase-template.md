# phase-N-*.md 共通テンプレート

各フェーズドキュメントは以下の節構成に統一する。ファイル名は `phase-<N>-<短い英語スラッグ>.md`（例: `phase-3-policy-env.md`）。

```markdown
# Phase N: <タイトル>

## 1. 目的

達成する価値を1〜3行 + overview.md の対応節（§番号）へのリンク。

## 2. スコープ

### 含むもの

このフェーズで実装する範囲を箇条書きで列挙する。

### 含まないもの（Non-goals）

「それは Phase M で行う」という形で、後続フェーズに送る範囲を明示する。スコープが曖昧だとフェーズ間で作業が重複したり漏れたりする。

## 3. 前提・依存

- 先行フェーズ: どのフェーズの完了が前提か。前提となる状態（「〇〇の呼び出し口が1箇所に集約されていること」など）まで具体的に書く。
- 並行可能フェーズ: 同時に進められるフェーズがあれば書く。
- 外部前提: 依存する外部ライブラリ・環境・決定事項があれば書く。

## 4. 実装方針

**コードは書かない。** どのファイル/クラス/モジュールに何の責務を置くかという設計判断のみを記述する。次のような粒度で書く:

- 「`path/to/foo.rb` に `Foo` を置く。責務は〜のみとし、〜には依存しない」
- 「既存の〜を〜に差し替える」
- 「〜は1箇所（例:モジュールのプライベートメソッド）に閉じ込め、次フェーズが安全に拡張できるようにする」

後続フェーズが積み上げやすいよう、拡張ポイントや契約（インターフェース）をここで明文化しておくと、フェーズ間の並行実装がリグレッションなく進められる。

## 5. 変更ファイル

- 新規: `path/to/new_file.rb`
- 変更: `path/to/existing_file.rb`（変更内容の要約）
- テスト: 新規/変更するテストファイルを具体的に列挙する
- ドキュメント: 更新するドキュメントがあれば列挙する（なければ「なし」と明記）

## 6. 受け入れ条件

checkbox形式。プロジェクト共通の条件（例:「CIが緑」）は全フェーズに含める。

- [ ] <検証可能な条件1>
- [ ] <検証可能な条件2>
- [ ] CI（テスト + lint）が緑。

## 7. テスト観点

- **単体**: 何を、どう検証するか
- **結合**: フェーズを跨いだ検証が必要な場合
- **エッジケース**: 見落としがちな境界条件を具体的に列挙する

## 8. リスク・決定事項

このフェーズで確定させる overview.md の決定事項（該当がなければ「なし」と明記し、理由を添える）。

リスク: このフェーズ特有のリスクと、その軽減策。

## 9. 参照

- overview.md の関連節（§番号）
- 参照する既存ファイルのパス

---

> **本ドキュメントは生きた文書である。** このフェーズの詳細化・実装のどちらの段階であっても、記載内容と食い違う判断をしたら、その場でこのファイルを書き換える（実装完了後にまとめて直すのではない）。前提・スコープ・実装方針が変わり `overview.md` の決定事項や他フェーズの前提に影響する場合は、`overview.md` / `README.md` / 影響する他の `phase-N-*.md` も合わせて更新する。変更した箇所には理由を一言添える（例:「〜だったため方針変更。詳細は§8参照」）。
```

## 書き方のポイント

- **「実装方針」は設計判断であって実装コードではない**。ここにコードを書き始めると、実装フェーズで実際に書いたコードとの間ですぐに乖離が生じる。責務配置とインターフェース契約に留める。
- **「参照」は overview.md の節番号を指し、内容を再掲しない**。仕様がoverview.mdとフェーズドキュメントの2箇所に書かれると、更新時に片方が古いまま残るリスクがある。
- **フェーズ内でさらに分割したくなったら、それは2フェーズへの分割のサイン**。1フェーズ1ファイルの原則を保つ。
- **末尾の「生きた文書である」の注記は削除・省略しない**。このドキュメント群の価値は「実装が終わった後も現状を正しく反映していること」にある。着手前のプランモードでの詳細化と、実装中に生じた見直しの両方でこのファイル自体を更新させることが、`overview.md`をSSOTとして機能させ続けるための唯一の担保になる。
