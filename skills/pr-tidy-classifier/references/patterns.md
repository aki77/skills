# T/NG パターン集

このファイルは、PR を整頓 (Tidy) と判断する **T1〜T14** と、人間レビューが必要と判断する **NG1〜NG7** の詳細を定義する。各パターンには「含まれる例」「含まれない例」「判定時のチェックポイント」を併記する。

判定時はこのファイルを読み込み、diff のハンク単位でどのパターンに該当するか確認すること。判断に迷う場合は `判定不能` に倒してよい (SKILL.md 参照)。

## パターンの出典について

Kent Beck `Tidy First?` に定義された整頓パターンは **言語中立な概念** として記述されており、特定の技術スタックに依存しない。本ファイルはその精神を踏まえつつ、実務でよく遭遇するパターンを整理したもの。

- **書籍準拠パターン**: T4/T5/T7/T8/T11/T12/T14 は書籍の概念に直接対応
- **実務拡張パターン**: T1/T2/T3/T6/T9/T10/T13 は書籍の精神に基づく実務的な拡張。特定技術名が例として登場するが、概念自体は汎用的

ベース: Kent Beck `Tidy First?` および [Findy Tech Blog: AIによるプルリクエスト分類器の作成 (2026)](https://tech.findy.co.jp/entry/2026/05/08/100000)。

## Tidy パターン (T1〜T14)

すべてが下記のいずれかに該当するなら、人間レビュー不要 (`Approve可`)。

### T1: ファイル移動・リネーム

- 含まれる例: ファイル/ディレクトリの場所変更、ファイル名のリネーム、それに伴う import パス更新
- 含まれない例: 移動と同時にファイル内のロジックも書き換えている、移動先で公開範囲が変わる (private→public 等)
- チェックポイント: 移動前後のファイル内容が機械的に対応しているか (diff の `-` と `+` がほぼ同一か)

### T2: 依存関係の整理 (import / require / use 等)

**実務拡張パターン** — 書籍の Dead Code / Normalize Symmetries の精神を import 管理に適用したもの。

- 含まれる例: モジュール参照の並べ替え・グルーピング、未使用 import/require/use の削除、公開インターフェース (barrel export, `index.ts`, `mod.rs` 等) の整理
- 含まれない例: 参照先モジュールを別のものに差し替えている (依存関係そのものが変わる)
- チェックポイント: 実体の依存関係 (どのモジュールに依存しているか) が変わっていないか

### T3: 型定義の移動・分離

- 含まれる例: `type` / `interface` / Ruby の `Sig` 等の定義を別ファイルに切り出す、共通型として共有
- 含まれない例: 移動と同時に型の構造を変える (フィールド追加・型変更)
- チェックポイント: 型の構造そのもの (フィールド・ジェネリクス) が同じか

### T4: コンポーネント・関数の分割・統合・抽出

- 含まれる例: 長い関数の一部を抽出、似た関数の統合、コンポーネントの分割。**振る舞いを変えない範囲で** の操作
- 含まれない例: 抽出ついでにロジックを変更、引数や戻り値の意味を変える
- チェックポイント: 抽出/統合前後で、外部から見た入出力が同じか

### T5: 変数・関数・props のリネーム

- 含まれる例: 関数名・変数名・props キーの変更、それに伴う spec/モック/呼び出し側の機械的更新、インラインスナップショット内の名前更新
- 含まれない例: リネームと同時に引数追加・型変更・挙動変更が混ざっている
- チェックポイント: 旧名→新名の対応が 1:1 で機械的に説明できるか

### T6: ESLint / Prettier / RuboCop / 他フォーマッタ修正

- 含まれる例: lint ルール違反の修正 (`prefer-const`, `no-unused-vars` など)、自動フォーマット適用、quote/indent 統一
- 含まれない例: lint 修正と称してロジックを書き換えている
- チェックポイント: 適用された lint ルール/フォーマッタが特定でき、その範囲内に収まっているか

### T7: 不要コード・ファイル削除

- 含まれる例: dead code の削除、使われていない関数・変数・ファイルの削除、コメントアウトされたコードの除去
- 含まれない例: 削除した処理の代替が新規追加されている (= 振る舞いが変わっている)
- チェックポイント: 削除対象が呼び出し元から本当に到達不可能か (静的解析で確認できるか)

### T8: 非推奨 API の機械的移行 (1:1対応)

- 含まれる例: `expect(x).toBeTrue()` → `expect(x).toBe(true)` のような assertion 置き換え、deprecated メソッドの完全互換な後継への置換
- 含まれない例: 移行と同時に挙動が微妙に変わる (例: 旧APIが暗黙の型変換していた、新APIは厳密比較)
- チェックポイント: 移行ガイドや公式ドキュメントで「等価な置き換え」と明記されているか

### T9: コード生成ツールの実行結果

**実務拡張パターン** — ツールが決定論的に出力する成果物の更新は、人間が目で追う必要のない機械的変更と見なせる。

- 含まれる例:
  - スキーマ・定義ファイル更新を契機とした生成ファイル (`*.generated.*`, `*_pb.go` 等) の更新
  - コード生成ツールの実行に伴い、手書きのクエリや宣言ファイルに**フィールドを機械的に追記するだけ**の変更 (ロジックや絞り込み条件は変えない)
  - 例: GraphQL codegen, gRPC protobuf, OpenAPI generator, i18n 自動生成 等
- 含まれない例: 生成物に手でロジックを書き加えている、生成元のスキーマ/定義自体を変えている、クエリの絞り込み条件・変数・引数を変えている
- チェックポイント: 「`generated/` 配下」「ヘッダコメントに `DO NOT EDIT` / `AUTO-GENERATED` の記載」など生成物を示す手がかりがあるか。手書きファイルへの変更は「生成ツールが要求する追記のみか」を確認する

### T10: スキャフォールド・雛形生成ツールの実行結果

**実務拡張パターン** — CLI ツールが決定論的に生成する雛形ファイルは、出力内容がツールの仕様によって定まるため、人間レビューより生成コマンドの妥当性確認が本質。

- 含まれる例: フレームワーク標準の scaffold / generator コマンドで出力された雛形そのもの (例: Rails generator, Nx generator, Django management command 等)
- 含まれない例: 生成後にビジネスロジックを手で書き加えている、generator の設定自体をカスタマイズしている
- チェックポイント: PR 説明文やコミットメッセージに生成コマンドが記載されているか、生成物の構造がそのツールの典型的な出力パターンに収まっているか

### T11: ガード節への変換

- 含まれる例: `if x { ... } else { return }` → `if !x { return }; ...` 形式への変換、ネスト深い `if` の早期リターン化
- 含まれない例: ガード節化と同時に条件式を変えている、return される値を変えている
- チェックポイント: 変換前後で実行パスが完全に等価か (真理値表で同じか)

### T12: 説明変数・定数の導入

- 含まれる例: マジックナンバーの定数化、複雑な式の中間変数への抽出
- 含まれない例: 抽出と同時に値を変えている、共通定数として広く参照されるが値が以前と微妙に違う
- チェックポイント: 抽出前後で計算結果が完全に同じか

### T13: Feature Flag 名の追加 (フラグ参照のみ)

- 含まれる例: 新しい Feature Flag のキー定義追加 (デフォルト値が既存挙動と一致する形)、フラグ判定の追加でも分岐先が既存実装のまま
- 含まれない例: フラグの後ろに新しい挙動を実装している
- チェックポイント: フラグが OFF のときの挙動が完全に既存と同じか

### T14: コメントの追加・修正・削除

- 含まれる例: コメント文の追加、誤字修正、古いコメントの削除、JSDoc/RDoc などのドキュメンテーション
- 含まれない例: コメント "アノテーション" がランタイム挙動に影響する (例: `# frozen_string_literal: true` の付け外しなど、実行に影響するもの)
- チェックポイント: 変更がプログラムの実行結果に影響しないか

### T15: 依存ライブラリのパッチバージョンアップ

**実務拡張パターン** — パッチバージョン (`x.y.Z`) の変更はバグ修正・セキュリティ修正のみであり、セマンティックバージョニングの規約上 後方互換性が保証される。Dependabot・Renovate によるパッチ更新 PR の大半はこれに該当し、よっぽどのことがない限り Approve可と判定する。

- 含まれる例:
  - `Gemfile.lock`, `package-lock.json`, `yarn.lock`, `Cargo.lock` 等のロックファイルで、バージョン番号のパッチ部分 (第3セグメント) のみが上がっている変更
  - セキュリティアドバイザリ (GHSA / CVE) 対応の Dependabot / Renovate PR
- 含まれない例:
  - マイナーバージョン (`x.Y.z`) やメジャーバージョン (`X.y.z`) が変わっている (→ NG7)
  - ロックファイル以外のファイルも同時に変更されている (コードや設定の追加修正が含まれる場合は個別に判定)
- チェックポイント:
  1. バージョン変更がパッチ部分のみか確認する (例: `1.19.2 → 1.19.3` は T15、`1.19.3 → 1.20.0` は NG7)
  2. 変更ファイルがロックファイルのみか確認する
  3. 上記2つが満たされれば `Approve可`
- 「よっぽどのこと」で NeedsReview に倒してよいケース:
  - リリースノートに「後方互換性のない変更が含まれる」と明記されている
  - Ruby インタープリタ自体・OS パッケージ等、ランタイムコアに近いライブラリ

## NG パターン (NG1〜NG7)

ひとつでも該当するハンクがあれば `NeedsReview`。

### NG1: 新しい条件分岐の追加

- 含まれる例: 新規 `if` / `case` / 三項演算子の追加、既存ロジックに新規分岐を挿入
- 注意: ガード節への変換 (T11) とは区別する。実行パスの真理値表が変わるかで判別する
- なぜNG: 分岐が増えると振る舞いが変わるため、テストカバレッジと意図の確認が必須

### NG2: 新しいビジネスロジック

- 含まれる例: 新しい計算式、新しいバリデーション、新しい状態遷移、新しい外部API呼び出し
- なぜNG: 仕様判断が必要なため、人間 (特にドメイン理解者) のレビューが必須

### NG3: API変更 (シグネチャ・契約の変更)

- 含まれる例: 関数の引数追加・削除・型変更、戻り値の型変更、HTTP API のリクエスト/レスポンス形式変更
- 含まれない例: 純粋なリネーム (T5)、内部実装の変更だけで外部契約が同じ
- なぜNG: 呼び出し側との整合・後方互換性の確認が必要

### NG4: 状態管理の変更

- 含まれる例: Redux/Pinia/Vuex などの store 構造変更、Rails のモデル属性追加・削除、グローバル状態の追加・削除
- なぜNG: 永続化・初期化・マイグレーション・他コンポーネントとの整合性の確認が必要

### NG5: テストの追加

- 含まれる例: 新規テストケースの追加、新規 spec ファイルの作成、テスト対象の網羅範囲を広げる変更
- 含まれない例: 既存テストの機械的なスナップショット更新 (T5/T8 等の波及)、リネームに伴う spec 内のシンボル更新
- なぜNG: 何をテストすべきと判断したかの妥当性を人間が確認する必要がある
- 注意: T5 リネームに伴ってインラインスナップショットが書き換わるケースを NG5 と誤認しないこと。波及元 (リネーム) と一緒に見ること

### NG6: UI 変更 (見た目・体験の変化)

- 含まれる例: コンポーネントの見た目を変えるスタイル変更、レイアウト変更、文言の意味が変わる修正、新規UI要素の追加
- 含まれない例: クラス名のリネーム (T5) で見た目は変わらないケース、コメント・JSDocのみの修正
- なぜNG: ユーザー影響が直接出るため、視覚的な確認・デザイナーレビューが必要

### NG7: 設定値の変更

- 含まれる例: 環境変数のデフォルト値変更、Feature Flag のデフォルト ON/OFF 切り替え、CIワークフローの実質的な動作変更、依存ライブラリのマイナー/メジャーバージョンアップ
- 含まれない例: コメントの追加 (T14)、フォーマット修正 (T6)、**依存ライブラリのパッチバージョンアップのみ** (→ T15 として Approve可)
- なぜNG: 環境間の挙動差・運用への影響評価が必要

## 判定の優先順位 (パターンが重なる場合)

ひとつのハンクが複数パターンに見えるとき:

1. **NG パターンが当てはまるなら必ず NG として扱う**
2. T パターンの中で迷った場合、より具体的なほうを選ぶ (例: T5 リネームと T8 非推奨API移行が両方マッチするなら、変更の主旨に近いほう)
3. 複数の T パターンが同居するPRは問題ない (例: T1 移動 + T5 リネーム の混合)

## 判定不能に倒すべきケース

- diff の意味が読み取れない (生成物っぽいが生成元が不明、独自DSL、バイナリ)
- 「振る舞いが変わらない」を確信できない (リフレクション、メタプログラミング、動的呼び出しが絡む)
- T パターンに見えるが、対象領域のテストが薄く、誤判定時の影響が大きい
- 30ファイル以下なのにサンプリング判断が必要なほど多種多様な変更が含まれる
