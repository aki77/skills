# 大規模PRのサンプリング手順

変更ファイル数が **31 ファイル以上** の場合、全ハンクの精査ではなくサンプリングで判定する。これは、「同じ作業の繰り返し」がAIには機械的に分析しやすい一方、無駄に時間とトークンを消費するため。Findy 記事の「大規模 PR のフォールバック」を踏襲する。

## サンプリング方針

### 1. パターン仮説を立てる

- 変更ファイルのリスト (`gh pr view --json files` または `git diff --stat`) と、最初の数ハンクから「これは何のリファクタか」を推測する
- 例: 全ファイルが `*.spec.ts` で同じ assertion 関数の置き換えなら T8 の仮説が立つ
- 例: 半数のファイルがあるディレクトリから別ディレクトリへ移動しているなら T1 の仮説が立つ

### 2. 代表的なファイルを 3〜5 件選んで精査

- パターンを代表すると思われるファイルを選ぶ
- ファイルサイズ・変更行数・拡張子のバラエティをある程度カバーする
- diff 全体ではなく、選んだファイルの diff を読み込んで T パターンと整合するか確認

### 3. 波及先ファイルを**セットで**含める (ここが最重要)

T パターンを特定したら、その変更によって**機械的に影響を受けるファイル**を最低 1 件サンプリング対象に加える。

| T パターン | 典型的な波及先 |
|-----------|---------------|
| T5 (リネーム) | `*.spec.{ts,rb,js}`、モック、フィクスチャ、インラインスナップショット、`__snapshots__/` 配下 |
| T1 (ファイル移動) | `index.ts` の re-export、import パスを更新した呼び出し側ファイル |
| T8 (非推奨API移行) | テストファイル全般、モック内のシグネチャ |
| T9/T10 (生成物) | 生成元のスキーマファイルや generator 設定、生成ツールが要求する追記を受けた手書きクエリ・宣言ファイル（フィールド追記のみの場合） |

**波及先を見落とすと、波及先のファイル**だけ**を見たときに NG パターンと誤認しやすい**。たとえば T5 リネームに伴うインラインスナップショット更新を見ると、テストの assertion が変わっているように見えて NG5 と誤判定してしまう。波及元 (リネーム) と波及先 (snapshot 更新) を**ペアで** 見ることで、機械的な追従であることが確認できる。

### 4. 「同一パターンか」の確認

選んだサンプルが**すべて同じパターン**で説明できるなら、残りのファイルも同様と推定して `Approve可` 寄りに判定してよい。サンプル間で異質な変更があれば、その異質ファイルをさらにサンプリングして見る。最終的に NG パターンが見つかれば `NeedsReview`。

### 5. 報告に明記する

サンプリング判定したことと、確認したファイル名を必ず報告に含める。例:

```markdown
- **サンプリング**: 実施
  - 確認したファイル (5/120):
    - `src/foo/old-name.ts` (T5 リネーム元)
    - `src/foo/old-name.spec.ts` (T5 波及: spec 内のシンボル更新)
    - `src/foo/__snapshots__/old-name.test.ts.snap` (T5 波及: スナップショット更新)
    - `src/index.ts` (T5 波及: re-export の更新)
    - `src/bar/uses-foo.ts` (T5 波及: 呼び出し側の import 名)
  - すべて同一パターン (`oldName` → `newName` のリネーム) として整合
```

これにより、後から「このサンプリングは妥当だったか」を人間が検証できる。

## 限界

- 100ファイル超のように非常に大きい場合でも、波及先サンプリングを怠ると誤判定する
- 「90% は T5、10% は別パターン」のような混在 PR は、最初の数ファイルだけ見て判定すると残り 10% を見落とす。サンプル選択時にディレクトリ構造や拡張子の偏りに気を配ること
- 確信が持てない場合は `判定不能` に倒すのが原則 (SKILL.md 参照)
